# 작업 요구사항 문서 (Requirements Document)

> **프로젝트명**: `<프로젝트명 입력>` — 생성하신 프로젝트 이름으로 교체하세요.
> **문서 버전**: v0.1
> **최종 수정일**: 2026-07-12
> **작성 목적**: LangGraph 기반 Gmail 읽기·삭제(휴지통 이동) 챗봇의 구현 범위, 기능/비기능 요구사항, 설계 제약을 정의한다.

---

## 1. 개요

사용자의 자연어 질문에 답하기 위해 Gmail을 **읽고**, 필요 시 특정 메일을 **휴지통으로 이동**하는 대화형 챗봇을 구축한다. 삭제는 자동으로 수행하지 않으며, 삭제 대상 메일을 사용자에게 먼저 제시하고 명시적 승인을 받은 뒤에만 실행한다. 승인 단계에서 사용자는 (1) 허용, (2) 금지, (3) 추가 의견 제출 중 하나를 선택할 수 있다.

핵심 설계 원칙은 **영구 삭제 불가**다. 모든 "삭제"는 되돌릴 수 있는 휴지통 이동(30일 복구 창)으로만 구현하며, 영구 삭제 경로는 도구 레벨과 OAuth 스코프 레벨에서 이중으로 차단한다.

---

## 2. 목표 및 범위

### 2.1 In Scope

- Gmail 메일 검색 및 조회 (제목·발신자·본문·라벨·날짜 기준)
- 메일 휴지통 이동 (`messages.trash` 상당 동작)
- 삭제 전 Human-in-the-Loop(HITL) 승인 플로우
- 승인 옵션 3종: 허용 / 금지 / 추가 의견
- 챗봇 응답의 실시간 스트리밍(서버 → 브라우저)

### 2.2 Out of Scope

- 메일 **영구 삭제** (`messages.delete`, `batchDelete`) — 명시적으로 금지
- 메일 발송(send), 초안 작성/수정, 답장/전달
- 라벨 생성·수정, 필터, 부재중 응답, 설정 변경
- 첨부파일 다운로드·처리 (초기 버전 제외)
- 다중 메일 계정 동시 연결 (초기 버전은 계정 1개 기준, 멀티테넌트는 향후 확장)

## 3. 기술 스택 (확정)

| 구분 | 선택 | 비고 |
|------|------|------|
| 언어 | Python 3.12 | langchain-mcp-adapters는 3.11+ 요구, 여유 충족 |
| 에이전트 프레임워크 | LangChain v1 / LangGraph v1 | `create_agent`, `HumanInTheLoopMiddleware` 사용 |
| 데이터 검증 | pydantic v2 | LangChain 1.x 네이티브 |
| LLM | ChatOpenAI (OpenAI 모델) | MCP 어댑터는 모델 비종속 |
| MCP 연동 | langchain-mcp-adapters (`MultiServerMCPClient`) | MCP 도구 → LangChain 도구 변환 |
| Gmail MCP 서버 | **trash 도구를 노출하는 서드파티/커뮤니티 서버** | 공식 Google Gmail MCP는 사용 불가 (§7.1 참조) |
| MCP transport | Streamable HTTP (원격) / stdio (로컬 개발) | SSE transport는 deprecated이므로 미사용 |
| 프론트엔드 스트리밍 | SSE (서버 → 브라우저) | 챗봇 응답 토큰 스트리밍 용도 |
| 상태 지속성 | PostgresSaver (프로덕션) | HITL 인터럽트에 필수 |
| 패키지·프로젝트 관리 | uv | |

> **주의**: "SSE"는 이 프로젝트에서 **프론트엔드 스트리밍(서버→브라우저)** 용도로만 사용한다. MCP 서버 연결 전송 방식으로는 SSE를 쓰지 않으며 Streamable HTTP를 사용한다. (SSE transport는 MCP 스펙 2025-03-26에서 deprecated됨)

## 4. 시스템 아키텍처

```
[브라우저 UI]
   │  ▲
   │  │  SSE 스트림 (토큰 이벤트 / 인터럽트 이벤트)
   ▼  │
[백엔드 서버 (FastAPI 등)]
   │        └─ 세션/스레드 ID 관리, Command(resume=...) 재개 처리
   ▼
[LangGraph Agent]
   ├─ ChatOpenAI (LLM)
   ├─ HumanInTheLoopMiddleware  ← trash 도구만 승인 게이팅
   ├─ Checkpointer (Postgres) ← 인터럽트 상태 지속
   └─ Tools (via langchain-mcp-adapters)
         │
         ▼  Streamable HTTP
   [Gmail MCP 서버]
         ├─ 읽기 도구 (search, get_message)  ← 노출
         ├─ trash 도구 (휴지통 이동)          ← 노출
         └─ delete/batchDelete 도구           ← 미노출 (제외)
         │
         ▼  OAuth (scope: gmail.modify)
   [Gmail API]
```

## 5. 기능 요구사항 (Functional Requirements)

### FR-1. 메일 읽기

- **FR-1.1** 사용자는 자연어로 메일을 검색·조회할 수 있어야 한다. (예: "지난주 미확인 프로모션 메일 보여줘")
- **FR-1.2** Gmail 검색 문법(from:, subject:, label:, after: 등)을 활용해 조건 검색을 지원한다.
- **FR-1.3** 개별 메일의 발신자·제목·날짜·본문·message_id를 조회할 수 있다.
- **FR-1.4** 읽기 계열 도구는 승인 없이 자동 실행된다(auto-approve).

### FR-2. 메일 삭제 (휴지통 이동)

- **FR-2.1** 삭제는 **휴지통 이동으로만** 수행한다. 영구 삭제는 어떤 경로로도 허용하지 않는다.
- **FR-2.2** 휴지통으로 이동된 메일은 Gmail의 30일 복구 창 내에서 복원 가능하다.
- **FR-2.3** 에이전트는 FR-3의 승인을 통과하기 전에는 절대 trash 도구를 실행하지 않는다.
- **FR-2.4** 영구 삭제 도구(`delete`, `batchDelete`)는 에이전트 도구 목록에 등록하지 않는다.

### FR-3. Human-in-the-Loop 승인

- **FR-3.1** trash 도구 호출 직전, 실행이 일시 중단되고 삭제 대상 메일 목록이 사용자에게 제시되어야 한다.
- **FR-3.2** 제시 정보에는 각 메일의 발신자, 제목, 날짜(가능하면 요약)가 포함되어야 한다.
- **FR-3.3** 사용자는 다음 세 가지 중 하나를 선택할 수 있다:

  | 옵션 | 사용자 행동 | 시스템 동작 | LangGraph 결정 타입 |
  |------|-------------|-------------|----------------------|
  | 1 | **허용** | 제시된 메일을 휴지통으로 이동 | `approve` |
  | 2 | **금지** | 삭제를 취소하고 실행하지 않음 | `reject` |
  | 3 | **추가 의견** | 사용자 텍스트를 에이전트에 전달, 재제안 유도 | `respond` |

- **FR-3.4** 옵션 3(추가 의견) 선택 시 사용자의 텍스트가 대화 맥락으로 반영되어 에이전트가 삭제 목록을 재구성할 수 있어야 한다.
- **FR-3.5** 승인 결과는 `Command(resume=...)`로 그래프에 전달되어 실행이 재개된다.
- **FR-3.6** (선택) 옵션 1 하위에 "일부만 승인"(제시된 N개 중 일부 선택 삭제) 기능을 둘 수 있다. 초기 버전 포함 여부는 별도 결정.

### FR-4. 스트리밍

- **FR-4.1** 챗봇의 텍스트 응답은 SSE로 브라우저에 토큰 단위로 스트리밍된다.
- **FR-4.2** SSE 스트림은 **"토큰 이벤트"**와 **"인터럽트(승인 대기) 이벤트"**를 구분해 전달해야 한다.
- **FR-4.3** 인터럽트 이벤트 수신 시 프론트엔드는 승인 UI(옵션 1/2/3 + 삭제 목록)를 렌더링한다.

## 6. 비기능 요구사항 (Non-functional Requirements)

### NFR-1. 보안

- **NFR-1.1 스코프 최소화**: OAuth 스코프는 `gmail.modify` **하나만** 발급한다. 이 스코프는 읽기 + 휴지통 이동을 모두 포함하되 영구 삭제(`https://mail.google.com/` 전체 스코프)는 포함하지 않는다.
- **NFR-1.2 도구 필터링**: MCP 서버가 영구 삭제 도구를 노출하더라도, 에이전트에는 읽기 + trash 도구만 로드한다.
- **NFR-1.3 이중 방어**: NFR-1.1과 NFR-1.2를 함께 적용해, 영구 삭제가 시도되더라도 (a) 도구가 없어서 호출 불가하거나 (b) 스코프 부족으로 API가 403을 반환하도록 한다.
- **NFR-1.4 프롬프트 인젝션 대비**: 메일 본문은 신뢰할 수 없는 입력으로 취급한다. 본문 내 숨겨진 지시가 도구 호출을 유도하지 못하도록, 노출 도구를 읽기+trash로 좁게 유지하고 시스템 프롬프트에서 본문 지시를 따르지 않도록 명시한다.
- **NFR-1.5 자격증명 보호**: OAuth credential 및 토큰 파일은 저장소·동기화 폴더에 커밋하지 않는다.

### NFR-2. 상태 관리 및 안정성

- **NFR-2.1 체크포인터 필수**: HITL 인터럽트/재개를 위해 checkpointer가 반드시 구성되어야 한다. 개발은 InMemorySaver, 프로덕션은 PostgresSaver 사용.
- **NFR-2.2 스레드 식별**: 각 대화는 고유 thread ID로 관리하며, 재개 시 동일 thread ID로 호출한다.
- **NFR-2.3 미승인 스레드 만료(TTL)**: 승인 대기 상태로 방치된 스레드를 주기적으로 스캔해 임계값(예: 24시간) 초과 시 폐기 처리한다. 승인 대기는 무한 latency를 유발할 수 있으므로 필수.

### NFR-3. 성능·확장성

- **NFR-3.1**: 멀티 인스턴스 배포 시 상태 공유를 위해 Postgres 체크포인터를 사용한다.
- **NFR-3.2**: Gmail API 쿼터를 고려해 대량 삭제는 개별 호출 반복보다 배치 방식(TRASH 라벨 일괄 적용)을 우선 검토한다.

## 7. 주요 설계 결정 및 근거

### 7.1 공식 Google Gmail MCP를 사용하지 않는 이유

공식 Google Gmail MCP 서버는 Developer Preview 단계로, 검색·스레드 조회·초안 생성·라벨 관리 위주의 제한된 도구 집합만 제공하며 **trash/delete 도구가 없다.** 사용 스코프도 `gmail.readonly` + `gmail.compose`로, 휴지통 이동에 필요한 `gmail.modify`가 없다. 따라서 삭제(휴지통 이동) 요구사항을 충족할 수 없어, trash 도구를 노출하는 서드파티/커뮤니티 Gmail MCP 서버를 채택한다.

### 7.2 영구 삭제 대신 휴지통 이동만 지원하는 이유

영구 삭제는 되돌릴 수 없고, `https://mail.google.com/` 전체 접근 스코프를 요구하며 Google의 restricted-scope 검증 심사 대상이 된다. 휴지통 이동은 30일 복구 창을 제공하고 `gmail.modify`로 가능해, HITL 승인과 결합 시 안전망이 이중으로 작동한다. 사용자 실수나 오승인에도 복구 여지가 남는다.

### 7.3 SSE transport 대신 Streamable HTTP를 쓰는 이유

MCP의 SSE transport는 스펙 2025-03-26에서 deprecated되었고 최신 스펙은 하위 호환 안내만 남긴 상태다. 신규 구현은 Streamable HTTP가 표준이며, 인증·수평 확장·stateless 배포·재개 가능한 스트림을 지원한다. (프론트엔드 응답 스트리밍용 SSE는 이와 무관하며 그대로 사용한다.)

## 8. 리스크 및 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| MCP 서버가 영구 삭제 도구를 함께 노출 | 오용 시 데이터 영구 손실 | 도구 미로드 + 스코프 제한 (이중 방어) |
| 메일 본문 프롬프트 인젝션 | 의도치 않은 도구 호출 | 노출 도구 최소화, 본문 지시 무시 프롬프트 |
| 승인 대기 스레드 방치 | 체크포인터 적체, 리소스 낭비 | TTL 만료 작업 (NFR-2.3) |
| 서드파티 MCP 서버의 스키마 변경 | 도구 연동 중단 | 서버 버전 고정, 도구 목록 검증 단계 추가 |
| 잘못된 검색 조건으로 대량 메일 삭제 대상화 | 대량 오삭제 시도 | HITL 승인 화면에서 전체 목록·건수 확인 후 진행 |

## 9. 완료 기준 (Acceptance Criteria)

- [ ] 자연어로 메일 검색·조회가 동작한다.
- [ ] 삭제 요청 시 삭제 대상 목록이 승인 UI에 표시된다.
- [ ] 옵션 1(허용) → 대상 메일이 휴지통으로 이동한다.
- [ ] 옵션 2(금지) → 삭제가 실행되지 않는다.
- [ ] 옵션 3(추가 의견) → 사용자 의견이 반영되어 재제안된다.
- [ ] 휴지통 이동된 메일이 Gmail 휴지통에서 확인·복구 가능하다.
- [ ] 영구 삭제가 어떤 경로로도 실행되지 않음이 검증된다 (도구 부재 + 스코프 403).
- [ ] 챗봇 응답이 SSE로 스트리밍되고, 인터럽트 이벤트가 별도로 전달된다.
- [ ] 서버 재시작 후에도 승인 대기 중이던 스레드가 복원·재개된다.
